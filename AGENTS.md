# AGENTS.md

## Learned User Preferences

- Use Bun exclusively — `bun`, `bunx`, never `npm`, `npx`, `yarn`, or `pnpm`
- Follow Conventional Commits — atomic commits, imperative mood, no AI/tool/agent references or trailers (`Made-with: Cursor`, `Co-authored-by: AI`); always show a commit review report and wait for explicit approval before committing
- Apply DRY, KISS, and SOLID principles — prefer simpler state over nullable + guards; avoid overengineering (prefer simple fixed config over port-check scripts when complexity adds minimal value); extract complex inline casts or nested ternaries into named helper functions; disable overzealous lint rules at config level rather than scattering disable comments
- Review from a senior staff engineer / head-of-engineering perspective when asked; for editor plugin code, review as ProseMirror/Tiptap core author checking for performance issues and memory leaks
- Prefer `lucide-react` icons via `lib/icons.ts` barrel file, not local SVG wrapper components
- Use named exports only (no `export default` except where required by framework); title-case acronyms in PascalCase names — `TocSidebar` not `TOCSidebar`
- Avoid hydration mismatches — never use `useState(() => loadFromLocalStorage())`; use `useEffect` with a "ready" state flag; never nest `<button>` inside `<button>`
- Prefer margin-positioned layout (e.g. TOC in left margin) so main content stays centered
- Use shadcn/ui components from `components/ui/` for new UI — design tokens are shadcn CSS variables; `--tt-*` tokens remain only for editor node SCSS
- Keep `.cursor/skills/` and `.cursor/rules/` tracked in git — never ignore them
- Always verify changes by running them — open the app and screenshot for UI; run tests end-to-end for test code; fix issues before considering work done
- When writing specifications or design documents, keep them self-contained; update related docs after feature implementation to keep teams synchronized

## Learned Workspace Facts

- TinyDocy is a lightweight collaborative document editor and testbed/MVP for the docs.plus production editor; built on Tiptap 3, Yjs, and Next.js 16 (App Router) with Bun as runtime (version pinned in `.tool-versions` and `package.json`); persistence via Hocuspocus (WebSocket) + SQLite via `@hocuspocus/cli`; dev servers run via `make dev` (Makefile with `make -j2`); features are validated here for performance, reliability, and collaboration before integration into docs.plus; all code fully owned; README.md at root is the sole README (GitHub displays it; tests/ uses TESTING.md to avoid duplicate README conflict); no CHANGELOG.md or deployment-pm2-traefik guide — user does not want these files
- Code quality: Biome for formatting/linting; ESLint scoped to Next.js rules only (React 19 experimental hooks rules disabled); TypeScript strict mode; `noExplicitAny` and `noNonNullAssertion` warn in Biome; `a11y/useSemanticElements` off
- Git hooks via Husky: pre-commit runs lint-staged (Biome), commit-msg runs commitlint (config `.commitlintrc.json` — dot prefix required)
- Naming: kebab-case files, PascalCase components with title-case acronyms, camelCase hooks/utils; path alias `@/*` maps to root; `types/` for TS declarations, app types colocated; localStorage keys follow pattern `tinydocy-theme`, `tinydocy-tabs`, `tinydocy-folds-{documentId}`; default theme light (dark mode opt-in)
- Tab list: Y.Doc `"global-tabs"` via Hocuspocus (`useSyncedTabs`); Y.Array of `{ id, title, createdAt }`; Playground at index 0 (stays first, not draggable); `activeTabId` in localStorage; tab reorder via dnd-kit (horizontal only, `restrictToHorizontalAxis`); reorderTab must use same derived array (`ensurePlaygroundTab(deduplicateTabs(raw))`) as display for correct indices; `flushSync` in Y.Array observer so state updates before drop animation; use `arrayMove` from `@dnd-kit/sortable`; avoid `restrictToParentElement` (can cause wrong collision/drop target); close-all: confirmation via native `confirm()`, on batch failure one `alert()` per failure, button right of + (icon only); single close calls `DELETE /api/documents/[id]` before removing from Y.Array
- Utilities split by domain in `lib/`: `utils.ts` (cn via clsx + tailwind-merge, formatBytes for human-readable byte strings), `shortcuts.ts` (keyboard), `editor-utils.ts` (ProseMirror/Tiptap), `url-utils.ts` (sanitization), `icons.ts` (Lucide barrel)
- UI: shadcn/ui (base-nova style, Base UI primitives, Tailwind v4) in `components/ui/`; old `tiptap-ui-primitive/` fully replaced; custom `ToolbarButton`, `Toolbar`, `Spacer`; Base UI data-attributes have higher CSS specificity — use plain `<div>` for toolbar/popover dividers; editor content area uses shadcn typography; TOC sidebar: 240px width, heading-level typography hierarchy, fold-aware, active heading highlighted with brand-colored left border
- Document model: first node is always H1 title (enforced by TitleDocument extension with schema `heading block*`); each body H1 starts a new section/chapter; heading levels 1-6 only
- Document persistence: Hocuspocus (WebSocket) + SQLite via `@hocuspocus/cli`; `DELETE /api/documents/[id]` removes from SQLite on tab close; empty docs seeded in SimpleEditor when `ydoc.getXmlFragment('default').length === 0`
- Heading drag-and-drop: forked `@tiptap/extension-drag-handle` architecture with `@floating-ui/dom`, custom mouse events (no HTML5 DnD), section-level dragging; ProseMirror quirks: `view.nodeDOM(pos)` returns null under Yjs (use `view.dom.children[i]`), Tiptap React v3 re-parents `view.dom` after construction
- Heading fold/unfold: `Decoration.node()` for heading classes and content hiding, `Decoration.widget()` for 3D CSS crinkle accordion; nested folds suppressed by `outerFoldEnd`; fold state keyed by `data-toc-id` in localStorage; `canMapDecorations` helper shared with HeadingScale; `HeadingFoldMeta.set` supports `persist?: boolean` — filter-driven folds use `persist: false` to skip localStorage save; `view().update()` conditionally copies `prevFoldedIds`/`prevAnimating` only when state actually changed (avoids O(F) allocations per keystroke)
- Heading filter: delegates ALL folding to HeadingFold plugin (no duplicated crinkle/hiding code); `filterSections` returns `directMatchIds + ancestorIds + descendantIds` for full hierarchy tree visibility; preview typing updates `matchedSectionIds` live (preview takes priority over committed slugs for count); title H1 (index 0) always skipped in fold dispatch and `findAllSections`; filter is per-tab — URL params cleared on component unmount; TOC sidebar highlights preview matches via `previewMatchIds` prop; filter state in URL search params (`?filter=...&mode=or|and`); local-only; shortcut CMD/CTRL+SHIFT+F; floating panel UI (sticky anchor + absolute panel) in top-right; `computeOwnRange` prevents false matches from nested subsections; typing fast-path: `canMapDecorations` → O(1) return prev when inactive, O(log D) decos.map when active (skips `findAllSections` entirely — section count cannot change on content-only edits)
- Editor page boundary: `.editor-with-toc` has canvas background (`--tt-canvas-color`); `.simple-editor-content` has border, rounded corners, box-shadow, top/bottom margin, and `overflow: hidden`; fold crinkle extends edge-to-edge via negative margins clipped by parent overflow; mobile override removes border/shadow/margin for edge-to-edge display
- Testing: Vitest (unit/fuzz/stress in `tests/unit/`, jsdom env, `tests/setup.ts` mocks localStorage) + Playwright (E2E/soak in `tests/e2e/`, 3 projects: `chromium`, `soak` 40-min timeout, `yjs-soak` 15-min timeout); standalone Yjs load harness in `tests/load/`; shared helpers in `tests/helpers/` (env-parsers, perf-config, soak-config, report-writer); `window.__tiptap_editor` exposed in dev mode for Playwright (typed in `types/window.d.ts`); `window.__HOCUS_URL` overrides Hocuspocus WS URL; `window.__HOCUS_TOKEN` injects per-user identity for collaboration tests; use `bun run test` not `bun test` (latter invokes Bun's native runner); Playwright requires `Meta`/`Control` for modifier keys — `Mod` is ProseMirror-only; `test-reports/` and `test-results/` gitignored; full suite: run `make dev` first, use `CI=` for Playwright so it uses existing servers (avoids webServer conflicts); load harness requires Hocuspocus up, uses `WebSocketPolyfill` (type assertion for Node.js `ws`); local dev server sustains ~50 clients (300+ causes convergence failure); performance tests: `make test-perf` (single-user typing latency, `PERF_HEADINGS`, `PERF_SHAPE`), `make test-perf-collab` (multi-user concurrent typing on shared doc, `PERF_COLLAB_USERS`, `PERF_COLLAB_HEADINGS`, `PERF_COLLAB_SHAPE`); soak tests use `SOAK_*` env vars (soak-config); load harness supports `LOAD_*` env vars (CLI overrides env); Makefile serves as CLI for all test config; collaboration tests use `retries: 2` and precondition assertions for timing-sensitive sync
- Collaboration testing and document generators: `useYjsDocument` uses module-level `docCache` (Map with ref counting) to persist Y.Doc instances across tab switches; each browser context injects a unique token via `addInitScript` + `window.__HOCUS_TOKEN`; multi-user soak supports N dynamic users via `SOAK_USERS` env var (default 3) with sequential joining for reliable sync; `generateLargeDocument` produces realistic content per section — 3-5 prose paragraphs, bullet/ordered/task lists, code blocks, blockquotes (`richContent` defaults to `true`); test editor registers TaskList/TaskItem extensions; `VALID_BLOCK_NAMES` includes `taskList`/`taskItem`
